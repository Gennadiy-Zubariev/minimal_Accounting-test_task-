# Історія взаємодії з LLM

Цей файл фіксує, як Claude (LLM) використовувався при розробці, згідно з вимогою ТЗ.

## Підхід до роботи з LLM

Claude виступав як технічний напарник — для уточнення особливостей Streamlit,
SQLite, Docker, обговорення бухгалтерських концепцій і генерації шаблонного коду.
Архітектурні рішення приймалися мною після обговорення; весь код було вичитано
й адаптовано до реальних потреб проєкту.

## 1. Архітектура та модель даних

### Початкова структура проєкту

Обговорювали, як організувати код мінімального застосунку з подвійним
записом так, щоб було видно розділення логіки (ТЗ явно це оцінює),
але не перетворити маленьке завдання на великий enterprise-проєкт.

Фінальне рішення:

- `app/chart.py` — план рахунків та правила проведення (Дт/Кт для кожного типу операції);
- `app/database.py` — вся робота з SQLite (ініціалізація, CRUD, звіти);
- `app/services.py` — валідація + проміжна бізнес-логіка між UI і БД;
- `app/ui/` — Streamlit-шар: точка входу, компоненти, сторінки.

Сторінки винесені у окремі файли (`app/ui/pages/*.py`), щоб легше було
орієнтуватись і додавати нові.

### Модель контрагента

Спочатку контрагент мав поле `kind` (`customer` / `supplier`).
Після першого тестування виявилось, що одна й та сама фірма у реальному
бізнесі може одночасно бути і клієнтом, і постачальником. Обговорили
з Claude варіанти:

1. дозволити дубль `edrpou` з різним `kind`;
2. прибрати `kind` з моделі взагалі — роль визначається типом операції.

Вибрав варіант 2 як більш правильний з точки зору бухгалтерського
моделювання: контрагент — це одна сутність, а роль («клієнт у цій
операції» або «постачальник у цій операції») визначається типом операції
і рахунками у проводці (1100 — коли він клієнт, 2000 — коли постачальник).

Завдяки цьому в «Книзі партнерів» один рядок може мати і дебіторку, і
кредиторку одночасно — повна картина розрахунків з контрагентом.

## 2. Подвійний запис та початкові залишки

### Питання, яке обговорювали

Як правильно проводити початкові залишки (готівка на старті, борги
клієнтів, борги постачальникам), щоб дотримався принцип подвійного запису
(Дт = Кт)?

### Варіанти

1. окрема таблиця `opening_balances` з простими числами — порушує принцип
   єдиного джерела правди і подвійного запису;
2. окрема процедура без проводок — те саме;
3. звичайні операції спец-типу `opening_balance` у кореспонденції з
   технічним рахунком Капітал.

### Прийняте рішення

Додав до плану рахунків **3000 Капітал / Нерозподілений прибуток** як
парний рахунок для початкових залишків:

- Готівка 1000 (актив): Дт 1000, Кт 3000
- Дебіторка 1100 (актив): Дт 1100, Кт 3000
- Кредиторка 2000 (пасив): Дт 3000, Кт 2000

Всі сальдо в системі обчислюються з журналу проводок — це єдине джерело
правди. Початкові залишки та щоденні операції рівноправні: однакова
структура, однаковий механізм обчислення.

Усвідомлене спрощення: рахунок 3000 об'єднує роль «стартового капіталу»
і «нерозподіленого прибутку». У реальних системах це окремі рахунки з
процедурою «закриття року» — я цього не робив, бо ТЗ просило мінімальну
систему.

## 3. Streamlit-специфіка

### Проблема з `st.form` і Enter

Під час тестування виявилось: коли на сторінці кілька `st.form`
(наприклад на різних вкладках через `st.tabs`), натискання Enter у полі
однієї форми інколи тригерило сабміт сусідньої форми (яка навіть не
відображалась). Це баг/фіча Streamlit — усі вкладки рендеряться в DOM
одночасно, просто приховані CSS-ом.

Обговорили варіанти. Рішення: **прибрати `st.form` на сторінках реєстрації
сум і операцій** — там поля і кнопка звичайні, сабміт тільки по кліку.

Enter залишили тільки у формі додавання контрагента (всередині пікера) —
там одна форма, конфлікту нема, і Enter там справді зручний.

### Ключі віджетів і очищення форм

У Streamlit віджет «пам'ятає» своє значення через `key` у `session_state`.
Просто присвоїти `value=0` після збереження не скидає поле — потрібне
явне видалення ключа з `st.session_state` і `st.rerun()`.

Додав допоміжну функцію `_clear_keys()` у сторінках, яка очищає конкретні
ключі після успішного збереження.

### Динамічні ключі при зміні контрагента/типу

Окрема проблема: при зміні контрагента або типу операції в пікері
значення суми/опису не скидалось. Причина — той самий `key` продовжував
жити в `session_state`.

Рішення: зробити `key` полів **залежним** від id контрагента і типу
операції. При зміні параметра ключ стає новий, Streamlit вважає поле
новим віджетом і показує початкове значення.

Винятком залишив `st.date_input` — у нього статичний ключ, щоб дата
зберігалась між змінами типу (для серійного вводу операцій за один день).

### Пікер контрагента

Обговорювали три варіанти UI:

1. `text_input` + окремий `selectbox` (початкова версія) — незручно, два
   окремі поля;
2. searchable `selectbox` зі спец-пунктом «Додати нового» в кінці —
   краще, але кнопку додавання треба шукати в списку;
3. searchable `selectbox` + окрема видима кнопка «➕ Додати» справа —
   фінальне рішення.

Варіант 3 зручний: пошук відбувається в одному полі (вбудований у
`selectbox`), а додавання — через окрему помітну кнопку. Форма додавання
з'являється тільки при натисканні на кнопку — так на сторінці зазвичай
одна форма, що важливо для правильної роботи Enter.

### Автоматичне відкриття сторінки «Початкові залишки»

Для нового користувача з порожньою БД Панель безглузда (усі нулі).
Зробив логіку: якщо в БД немає жодної операції, `main.py` за замовчуванням
відкриває сторінку «Початкові залишки» з привітальною підказкою. Коли
БД вже не порожня — відкриває Панель.

Реалізація: `db.is_database_empty()` + початкове значення `session_state["nav"]`.

## 4. SQLite

### Чому SQLite, а не ORM (SQLAlchemy тощо)

Для мінімального застосунку голий SQL через `sqlite3` простіший і
читабельніший. ORM додав би шар абстракції без реальної користі на цьому
масштабі. SQLite вбудований у Python (не потрібно ставити окремо), база
— один файл, ідеально для volume в Docker.

### Контекстний менеджер `get_conn()`

Використав `@contextmanager` з `contextlib`, щоб код кожної функції з БД
не повторював обгортку `try/except/finally` з `commit/rollback/close`.
Це робиться один раз у `get_conn()`, далі скрізь: `with get_conn() as conn: ...`

Підключення створюється на кожен виклик — це безпечно для Streamlit, бо
одне sqlite-з'єднання не можна шарити між потоками.

### Enforcement foreign keys

SQLite за замовчуванням не перевіряє зовнішні ключі. Додав
`PRAGMA foreign_keys = ON` в `get_conn()`, щоб `ON DELETE CASCADE` для
`journal_entries` справді працював.

### Безпека від SQL-injection

Всі параметри запитів передаю через `?`-плейсхолдери, а не склеюю рядки.
Навіть у локальному додатку це хороша звичка.

## 5. Docker

### Структура

- `Dockerfile` — Python 3.11-slim як базовий образ, `pip install` з
  кешуванням (спочатку `COPY requirements.txt`, потім `COPY . .`), три
  змінні оточення Streamlit для headless-запуску.
- `docker-compose.yml` — один сервіс `accounting`, два volume:
  - `./data` — для БД SQLite (переживає перезапуски);
  - `./static/backgrounds` — для фонових картинок (можна додавати без
    перебудови образу).

### Фон інтерфейсу

Streamlit не має нативного API для фонового зображення. Обговорили
варіанти:

1. віддавати картинку як статичний файл через `enableStaticServing` —
   потребує додаткових налаштувань, проблеми з путями в Docker;
2. вставити base64-дані прямо в CSS як data-URI — нульові залежності,
   працює у будь-якому оточенні.

Вибрав варіант 2.

### Напівпрозорий UI

Картинка ставиться на `.stApp`, а блоки з текстом/таблицями — напівпрозорі
(`rgba(255, 255, 255, 0.75)` тощо). Співвідношення підбирав так, щоб
картинка проглядала, але текст лишався читабельним.

## 6. Валідація

### Формат ЄДРПОУ

Прийняв формальну перевірку: тільки цифри, 8 (юрособа) або 10 (ФОП) знаків.
Без обчислення контрольної цифри — це виходить за межі scope мінімального
застосунку.

### Дата операції

Додав `max_value=date.today()` у `st.date_input` (блокує вибір майбутньої
дати в UI) + перевірку в `services.register_operation()` (blok на рівні
сервісу, навіть якщо хтось обійде UI).

### Дублювання контрагентів

На рівні БД — `UNIQUE` на `edrpou`. На рівні сервісу — явна перевірка
перед вставкою з людським повідомленням про помилку замість низькорівневого
`IntegrityError`. Це «defense in depth».

## 7. Що свідомо НЕ зроблено

Ці рішення прийняті з огляду на пряму вимогу ТЗ «не будуйте забагато».

- ORM — голий SQL читається чіткіше;
- Міграції БД — для v1 достатньо `CREATE TABLE IF NOT EXISTS`. При зміні
  схеми (напр. коли прибирали поле `kind`) потрібне ручне видалення БД;
- Unit-тести — поза межами scope ТЗ;
- Автентифікація, ролі, дозволи;
- Мультивалютність, податки, узгодження (reconciliation);
- Редагування/сторнування операцій з UI;
- Закриття року (перенос чистого результату на рахунок Капітал);
- Деталізація плану рахунків (окремі рахунки під «Статутний капітал» і
  «Нерозподілений прибуток» тощо).

## 8. Бухгалтерські концепції

Клаудом також послуговувався для уточнення базових концепцій (рахунки
активів/пасивів/доходу/витрат, нормальне сальдо, кореспонденція Дт/Кт при
початкових залишках різних типів рахунків, рівняння балансу
`Активи = Пасиви + Власний капітал`). Це допомогло правильно збудувати
модель і пояснити логіку в коментарях коду.

## Уроки

- Streamlit любить `session_state` — ключі живуть між рендерами, і це
  треба явно враховувати при очищенні форм.
- `st.form` зручний, але кілька форм на сторінці можуть конфліктувати
  через Enter — якщо потрібен чіткий контроль, краще обійтись без нього.
- `@contextmanager` + `with` — правильний патерн для роботи з БД, файлами
  та іншими ресурсами в Python.
- У бухгалтерії принцип «єдине джерело правди — журнал проводок» дає
  величезну перевагу: не треба синхронізувати кілька місць, усі звіти
  обчислюються з одних даних.
- Коментарі-«карта» у верхній частині кожного файлу («ДЕ ПРАВИТИ ЩО»)
  значно пришвидшують орієнтування в коді.
